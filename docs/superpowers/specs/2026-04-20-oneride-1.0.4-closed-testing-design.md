## Goal

Produce a mobile+backend build stable enough to put in front of closed testers without them hitting:
- Money bugs (orphaned wallet deductions, duplicate withdrawals, lost referral credits)
- Lost trips (rides stuck "accepted" after driver disconnects, WS silent drop-out, 0,0 location, low-accuracy GPS)
- Broken profile edits (image upload that silently drops the file, crash on null `created_at`)
- Whole-app crashes (no error boundary, unguarded `.find()` on non-array, unmounted setState)

## Scope boundary

**In scope:** `mobile/` (user + rider screens), `backend/pkg/handlers/` and related Go files.

**Explicitly out of scope:**
- `admin/` web fixes (localStorage token, CSRF, CSV injection, stale cache) — separate spec
- Map library consolidation (`react-native-maps` vs Leaflet-in-WebView) — separate spec
- Full battery / adaptive-polling optimization (Batch E from the 2026-04-18 audit) — separate spec; we do land the Android foreground service in this release because it's a blocker for driver trip continuity
- Nice-to-haves: telemetry, haptics, a11y polish, bundle size

**Tester audience:** closed testers recruited by the project owner. Not public production. Known-issues section of the release notes is allowed.

## Workstreams

Each workstream starts with an **audit step** that confirms the gap still exists before patching. Commit `84267f9 fix(security): harden money flows, auth, and chat handlers` already touched several money paths, and we don't want to double-patch or regress.

### A. Money integrity (backend)

Objective: every wallet mutation is atomic and idempotent; no silent failures credit or debit without a paired record.

| # | Item | File & line (from audit) | Audit-first behavior |
|---|------|---|---|
| A1 | Wallet balance deducted but `wallet_transactions` row never saved | `backend/pkg/handlers/handlers.go:2861, 2985, 4561, 5055, 5119` | Confirm the transactions are now wrapped in a single DB transaction with the balance update. If already fixed in `84267f9`, document and move on. |
| A2 | Commission deduction errors silently logged; ride continues | `handlers.go:92-94, 108-110` | Either propagate as 5xx, OR queue for retry and surface via commission summary. Decide based on existing pattern. |
| A3 | Withdrawal submit has no idempotency key — double-tap pays twice | `handlers.go` withdrawal POST + `mobile/src/screens/Rider/RiderEarningsScreen.tsx:258-298` | Add `Idempotency-Key` header from mobile (UUID per submit attempt), backend dedupes at handler entry. |
| A4 | Promo `usage_count` race under concurrent claims | `handlers.go:787, 1552, 2241` | Use `SELECT ... FOR UPDATE` inside the claim transaction, or atomic `UPDATE ... WHERE usage_count < max_uses`. |
| A5 | Referral wallet credit without row lock — lost/double bonuses | `handlers.go:6530, 6538` | Same pattern as A4: wrap credit in transaction with lock on referrer's wallet row. |

**Verification per item:** read current state → write a targeted failing integration test where practical → patch → re-run. No change if the gap no longer exists.

### B. Ride-lifecycle reliability (rider mobile + backend)

Objective: a driver's trip doesn't disappear into a half-state when the network hiccups.

| # | Item | File | Fix |
|---|------|---|---|
| B1 | Driver goes offline, rides stay `accepted`, passenger sees phantom pickup | `mobile/src/screens/Rider/RiderDashboardScreen.tsx:365-375` + backend `OfflineDriver` handler | On explicit offline: cancel any `accepted`/`en_route` rides for this driver, notify passengers over WS, reset `is_available`. On WS disconnect (inferred offline): backend reaper — new background goroutine, 30s interval, cancels rides where driver last_ping > 90s. |
| B2 | WS reconnect hard-stops after 5 attempts — driver silently disappears | `RiderDashboardScreen.tsx:258-262` | Replace capped retry with exponential backoff: 1s, 2s, 4s, 8s, 16s, 30s, then stay at 30s indefinitely while app is foregrounded. Show "Reconnecting…" banner after 3 attempts. |
| B3 | GPS accuracy `Balanced` (~300m error) throughout — passengers see wrong dot | `RiderDashboardScreen.tsx:287` | Accuracy tier by state: `BestForNavigation` while `en_route`/`accepted`, `High` while `online` idle, `Balanced` while app backgrounded with no active ride. |
| B4 | No Android foreground service — driver app killed ~10min backgrounded, trip abandoned | `mobile/android/app/src/main/AndroidManifest.xml` + new service | Declare `FOREGROUND_SERVICE_LOCATION` permission; add a foreground service bound to active-trip lifecycle, persistent notification "ONE RIDE is tracking your trip". Start on `ride accepted`, stop on ride complete/cancel. |
| B5 | Location fallback sends `0,0` if GPS fails | `RiderDashboardScreen.tsx:318-353` | Already partially addressed in `08f6016`. Confirm: reject `0,0` and any coord outside PH bounding box; show error, don't POST. |

### D. Profile image upload (real fix — not a stub)

Currently `EditProfileScreen.tsx:245-250` picks an image, sets it locally, then drops it: `updateProfile` only sends `name/phone/profile_image=""`. No backend endpoint consumes the file. Selected image is lost on refresh.

**Backend:**
- New handler `UploadProfileImage` in `handlers.go`
- Route: `POST /user/profile/image` (authenticated) and `POST /rider/profile/image`
- `multipart/form-data`, field name `image`
- Validation: content-type ∈ `{image/jpeg, image/png, image/webp}`, max 5 MB, decode-and-reject-if-not-valid-image
- Storage: reuse whatever pattern payment proofs use (likely local disk under `uploads/` served via static route; confirm during audit step)
- On success: update `users.profile_image` with public URL; return `{ profile_image: "<url>" }`

**Mobile:**
- New utility `mobile/src/utils/upload.ts` — `uploadMultipart(endpoint, fileUri, fieldName, extraFields?)` using `fetch` with `FormData`
- `EditProfileScreen.handleSave`: if `newImageUri && newImageUri !== 'remove'` → call `uploadMultipart('/user/profile/image', newImageUri)` BEFORE the `updateProfile` call for name/phone; merge returned URL into `updateUser`
- Same pattern on rider-side edit profile screen (confirm exact path during audit — may be `mobile/src/screens/Rider/RiderProfileScreen.tsx` or equivalent)
- Progress indicator during upload, clear error message on failure (size/format)

## Architecture notes

### Audit-first discipline
Every workstream-A task begins with "read the current code, confirm the audited gap is still present." If `84267f9` already resolved it, the task output is a one-paragraph "verified fixed, no change needed" note in the implementation plan. This prevents re-patching already-hardened code.

### No new screens, no new navigation
All work lands in existing files. The only new files created:
- `mobile/src/components/ErrorBoundary.tsx` (Workstream C1)
- `mobile/src/utils/upload.ts` (Workstream D)
- Android foreground service declaration + Kotlin/Java service class (Workstream B4) — follow standard Expo foreground-service pattern

### Shared backend helper
Workstream D creates a single `uploadProfileImage(userID, roleTable)` helper in `handlers.go` used by both user and rider routes. Single validation path, single storage path.

### Testing approach
- **Backend:** `cd backend && go build ./... && go test ./...` after each backend task
- **Mobile:** TypeScript compile (`tsc --noEmit`) and a manual smoke pass on a real Android device per workstream. The user has an Android signing setup already (`946b0b6` commit).
- **No new automated-test infra.** Use existing patterns only.
- **Manual smoke pass** after all workstreams, before cutting the `.aab`:
  - Register → onboarding → book a ride → complete ride → rate → check wallet
  - Rider: register → apply → get approved → go online → accept ride → complete → withdraw
  - Profile: user edits name + phone + photo; rider edits name + phone + photo + vehicle
  - Network toggle mid-trip (airplane mode 10s, back on) — rider stays connected, trip persists

## Risks & mitigations

| Risk | Mitigation |
|------|---|
| Foreground service on Android triggers Play Store policy review delay | Use `FOREGROUND_SERVICE_LOCATION` with a visible persistent notification tied to active-trip only; this matches the documented allowed use case. Document the policy in the release notes. |
| Money-flow changes accidentally regress already-fixed paths | Audit-first step per task. Diff against the state before `84267f9` only via `git show`, not via re-implementation. |
| Image upload endpoint becomes an unauthed DoS surface | `requireAuth` middleware, 5 MB body limit, per-user rate limit (reuse existing rate-limit middleware from `824d14f`). |
| Release notes mention unfixed criticals from the audit, eroding tester trust | "Known issues" section is short and honest; list only the deferred items relevant to testers (admin web, map library consolidation are invisible to testers — not worth mentioning). |
| `.aab` build fails because the signing config changed | Test build locally before re-uploading; don't overwrite the Play Console draft until the new artifact is verified. |
